## Replacing Simmer Market with Polymarket Builder API

**Purpose:** This document maps every Simmer SDK/API dependency in the current codebase
to the equivalent Polymarket Builder API surface. Build these components in order.

## 1. What Simmer Currently Does For You (Summary)

| Role | Simmer abstraction | What it really is |
|------|-------------------|-------------------|
| Auth | `SIMMER_API_KEY` + `SimmerClient` | Wallet + L2 API key pair |

| Fast market discovery | `GET /api/sdk/fast-markets` | Already replaced by Gamma API in `market_utils.py` |

## 2. Polymarket Builder API — Key URLs & Auth

### Base URLs
```
Gamma (metadata/discovery): https://gamma-api.polymarket.com   ← already in codebase
CLOB (trading):             https://clob.polymarket.com
```

### Authentication — Two Layers

Polymarket uses a two-layer key system. You must implement both.

**L1 — Wallet (signing key)**
- An Ethereum EOA (externally owned account) private key
- Used to sign the one-time derivation request that produces an L2 key
- env var: `POLY_PRIVATE_KEY` (hex private key)

**L2 — Derived API key (for HTTP calls)**
- Generated by POSTing a signed message to `POST /auth/derive-api-key`
- Returns: `{ api_key, api_secret, api_passphrase }`
- All CLOB HTTP requests use these three values as headers
- env vars: `POLY_API_KEY`, `POLY_API_SECRET`, `POLY_API_PASSPHRASE`
- Optional: store `POLY_FUNDER` address (the wallet that holds USDC)

### COMPONENT 2 — Market Resolver (`poly_market.py`)

**Purpose:** Replaces `client.import_market(url)` which returned a Simmer `market_id`.

In direct Polymarket, you need TWO identifiers per market:
- `condition_id` — the Conditional Token Framework (CTF) condition (hex string)
- `token_id` — the specific YES or NO token (used for placing orders)

**What to build:**
```
Function resolve_market(slug) → { condition_id, yes_token_id, no_token_id, question, end_time }
  - Query Gamma: GET https://gamma-api.polymarket.com/events?slug={slug}
  - Extract conditionId from the market object
  - Query CLOB: GET https://clob.polymarket.com/markets/{condition_id}
  - Extract clobTokenIds[0] (YES) and clobTokenIds[1] (NO)
  - Cache result to market_id_cache.json (same mechanism as today)

Function get_token_id(slug, side) → str
  - Returns the correct token_id for "yes" or "no" side
```

**CLOB endpoint used:**
```
GET https://clob.polymarket.com/markets/{condition_id}
Response includes: { condition_id, tokens: [{token_id, outcome}], ...}
```

**Note:** The Gamma API call is already done in `market_utils.py`. You just need to add
the second CLOB call to get `token_id`, and then cache `{slug → {condition_id, yes_token_id, no_token_id}}`.

**Replaces in codebase:**
- `fast_trader.py:493–559` — `import_fast_market_market()` (no more import quota!)
- `fast_trader.py:311–325` — `warm_import_cache()` (still useful, but no rate limit concerns)
- `fast_trader.py:514–518` — import quota tracking (`_get_import_count_today()`)
- `data/daily_spend.json` — quota tracking file (no longer needed for imports)

**Key improvement:** No daily import quota. Resolve as often as needed.

### COMPONENT 3 — Order Execution (`poly_trade.py`)

**Purpose:** Replaces `client.trade(market_id, side, amount, source=...)`.

Polymarket CLOB supports two order types. For your fast-loop strategy, use **Market Order**
(immediately fills at best available price).

**What to build:**
```
Function execute_trade(token_id, side, amount_usdc, auth) → {
    success, order_id, shares_bought, average_price, error
}
  - Build a signed order (EIP-712 signature with wallet private key)
  - POST to CLOB /order endpoint
  - Return normalized result matching current execute_trade() return shape

Function build_market_order(token_id, side, amount_usdc, price_worst_case) → order_dict
  - Construct the order payload
  - Sign with L1 private key (EIP-712)
```

**CLOB endpoint used:**
```
POST https://clob.polymarket.com/order
Headers: { POLY_ADDRESS, POLY_SIGNATURE, POLY_TIMESTAMP, POLY_NONCE }
Body: {
  order: {
    salt: <random int>,
    maker: <your address>,
    signer: <your address>,
    taker: "0x0000000000000000000000000000000000000000",
    tokenId: <token_id>,
    makerAmount: <usdc amount in 6-decimal USDC units>,
    takerAmount: <shares expected>,
    expiration: <unix timestamp>,
    nonce: 0,
    feeRateBps: 0,
    side: 0 (BUY) or 1 (SELL),
    signatureType: 0,
    signature: <EIP-712 signature>
  },
  owner: <your address>,
  orderType: "FOK"   ← Fill Or Kill = market order equivalent
}
```

**Signing:** Requires EIP-712 typed-data signing with the `Exchange` contract domain.
Use the `py-clob-client` library (Polymarket's official Python client) to handle
order construction and signing, OR implement manually with `eth_account`.

**Replaces in codebase:**
- `fast_trader.py:579–595` — `execute_trade()` function
- `fast_trader.py:61` — `TRADE_SOURCE` (replace with `referralCode` field in order if needed)

### COMPONENT 6 — Trade History (`poly_history.py`)

**Purpose:** Replaces `GET https://api.simmer.markets/api/sdk/trades?limit=200&venue=polymarket`
used by `backfill_trade_outcomes()`.

**What to build:**
```
Function get_trade_history(auth, limit=200) → list of {
    id (order_id), condition_id, side, size, price, timestamp, status
}
  - GET /trade-history endpoint with auth headers
  - Map each fill to the shape that backfill_trade_outcomes() expects:
    { "id": order_id, "cost": proceeds_from_resolution, ... }
```

**CLOB endpoint used:**
```
GET https://clob.polymarket.com/trade-history?maker_address={address}&limit={n}
```

**P&L Calculation change:** Simmer's `/trades` returned a `cost` field (proceeds after resolution).
With direct Polymarket, you'll need to compute P&L differently:
- Winning position: `pnl = shares_held * 1.0 - amount_paid`
- Losing position: `pnl = 0 - amount_paid`
- Track `amount_paid` from order fills in trade-history

**Replaces in codebase:**
- `fast_trader.py:347–372` — `backfill_trade_outcomes()` function
- `fast_trader.py:349` — `https://api.simmer.markets/api/sdk/trades` URL

### COMPONENT 7 — Redeemer (`poly_redeem.py`)

**Purpose:** Replaces `client.redeem(market_id, side)`.

Redemption in Polymarket is an **on-chain transaction** — it calls the CTF Exchange smart
contract's `redeemPositions()` function on Polygon. This cannot be done via HTTP API alone.

**What to build:**
```
Function redeem_position(condition_id, side, auth, web3_provider) → { success, tx_hash }
  - Check if position is actually redeemable (market resolved, holding winning tokens)
  - Call CTF Exchange contract redeemPositions() on Polygon
  - Sign and broadcast transaction with L1 private key

OR — simpler approach first:
  - Check if CLOB has a redeem endpoint (some builder integrations do)
  - POST /redeem with { condition_id }
```

**On-chain details:**
```
Network:           Polygon Mainnet (chain_id=137)
CTF Exchange:      0x4D97DCd97eC945f40cF65F87097ACe5EA0476045
USDC (PolyUSDC):  0x2791Bca1f2de4661ED88A30C99A7a9449Aa84174
RPC:              https://polygon-rpc.com  (or Alchemy/Infura endpoint)
```

**Replaces in codebase:**
- `fast_trader.py:328–344` — `run_redeemer()` function
- `fast_trader.py:337` — `get_client().redeem(market_id=..., side=...)`

**Dependency needed:** `web3` library (or `py-clob-client` which wraps this)

## 5. Recommended Library

Polymarket publishes an official Python client that handles auth, order signing, and CLOB calls:

```
py-clob-client         # Polymarket's official Python CLOB client
```

Install:
```bash
pip install py-clob-client
```

This library provides:
- `ClobClient` — drop-in client with all CLOB methods
- `create_or_derive_api_key()` — handles L1→L2 key derivation
- `create_market_order()` — builds and signs orders
- `get_order_book()`, `get_last_trade_price()` — market data

If you use `py-clob-client`, Components 1, 2, 3, 4, 5, 6 map directly to its methods.
Component 7 (redeem) still requires a direct web3 call or a separate Polymarket utility.

**Add to `requirements.txt`:**
```
py-clob-client >= 0.14.0
web3 >= 6.0.0          # for on-chain redemption
eth-account >= 0.8.0   # for EIP-712 signing (dependency of py-clob-client)
```

**Remove from `requirements.txt`:**
```
simmer-sdk == 0.8.27   ← delete this line
```

## 9. Key Differences vs Simmer (Things to Handle)

| Topic | Simmer | Direct Polymarket |
|-------|--------|--------------------|
| Market ID | Simmer-specific opaque `market_id` string | `condition_id` (CTF hex) + `token_id` per outcome |
| Trade amount | USD amount → Simmer handles conversion | Must specify `makerAmount` in USDC (6 decimals) |
| Order type | Abstracted "trade" call | Explicit FOK/GTC/GTD order with signature |
| Slippage | Simmer handles internally | You set `price` field (worst acceptable price) |
| Rate limits | 6 imports/min, 10/day quota | No import quota — resolve freely |
| Redeem | REST call to Simmer | On-chain Polygon transaction |
| Auth | Single API key | Wallet + derived L2 key pair |
| P&L | `cost` field in Simmer trades response | Must compute from fill price × shares |
